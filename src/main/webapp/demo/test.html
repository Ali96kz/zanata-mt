Kernel panic at trace_find_cmdline() function

Solution In Progress - Mis à jour2014-08-12T02:11:46+10:00 -
English
  • No translations currently exist.

Environnement

  • Red Hat Enterprise Linux 6

Question

  • Kernel panic at trace_find_cmdline() function

Résolution

  • A private bug 1123102 has been opened and Engineering team is working on this issue.

Cause

There is insufficient checking in the trace code to be certain that the status of the ring buffer has not changed. If a read of the trace file is in progress and a trace_pipe interrupts it, there is a rare set of circumstances that can cause an illegal address reference.

Upstream commit 492a74f4210e15f4701422e2e1c4cd3c1e45ddae may be part of the fix. Additional corrections have been proposed in the LKML

  • There is at least one problem:
    if (pid > PID_MAX_DEFAULT) {
        strcpy(comm, "<...>");
        return;
    }
  • It should have caught the too-large pid value, since pid is an int, 0xea83905f looks like a negative value. It seems likely there is another problem, the pid value is wrong or the ent pointer is wrong.
